# 教授版摘要

## 1. 文件目的

本文件提供教授快速判斷：是否、如何、以及在什麼限制下，將 AI Agent Workflow 導入課堂作業或專題實作。重點不是推廣特定工具，而是建立一套可審查、可測試、可追溯的學習流程。

本資料包主張「Agent 進入 workflow」，不是「學生把作業交給 AI」。學生仍必須理解問題、設計架構、判斷取捨、驗證結果，並對最後提交內容負責。

## 2. 為什麼要導入 AI Agent Workflow

AI Agent 已逐漸出現在程式開發、文件整理、測試補強與除錯流程中。課堂若完全禁止，學生可能缺少面對實務工具的訓練；若完全放任，則會造成不可追溯、不可評分、也不利於學習的產出。

較務實的做法，是把 Agent 使用納入明確工作流：先 inspect、再 plan、經人批准後 patch，接著 test、review，最後由學生整理 commit。如此可讓學生學習如何指揮工具、檢查工具、限制工具，而不是依賴工具。

## 3. 核心主張：Human-led Architecture

Human-led Architecture 指的是：人類負責架構、需求、風險判斷、驗證與最終責任；Agent 只能在受限制的範圍內協助執行。

| 責任 | 應由誰主導 | 說明 |
| --- | --- | --- |
| 問題定義與需求拆解 | 學生 | 必須能說明要解決什麼問題，以及成功條件。 |
| 架構與 invariants | 學生 | 必須定義不能被破壞的資料格式、API contract、權限與安全規則。 |
| 修改建議與草稿 | Agent 可協助 | Agent 可提出 plan、patch、測試草稿或文件草稿。 |
| 驗證與取捨 | 學生 | 必須檢查 edge cases、測試結果與副作用。 |
| 最終提交 | 學生 | 不允許 Agent 自動 push main；提交前必須 review。 |

## 4. 學生仍然必須負責的能力

導入 Agent 不代表降低學習要求。相反地，學生需要更清楚地展現以下能力：

- 能把模糊需求轉成可執行任務。
- 能定義 invariants 與 edge cases。
- 能閱讀 `git diff`，判斷修改是否合理。
- 能建立或執行 test harness 驗證結果。
- 能用 prompt / XML 約束限制 Agent 的任務範圍與輸出。
- 能管理 context window 與 memory 風險，避免過期、錯誤或未審查資訊影響結果。
- 能說明哪些部分是自己決策，哪些部分是 Agent 協助產生。

## 5. Agent 適合協助的任務

Agent 適合放在明確邊界內使用，尤其是可以被人檢查與測試的工作。

| 任務類型 | 適合程度 | 條件 |
| --- | --- | --- |
| inspect 現有檔案與整理問題 | 高 | 不修改檔案，先回報觀察。 |
| 提出修改 plan | 高 | 需列出影響範圍、風險與驗證方式。 |
| 小範圍 patch | 中高 | 必須限制檔案範圍，修改後檢查 diff。 |
| 補測試與文件 | 中高 | 需由學生確認測試是否對準需求。 |
| 大型架構決策 | 低 | Agent 可提供選項，但不應取代人類判斷。 |
| 自動 push main | 不允許 | 應由學生或團隊 review 後手動處理。 |

## 6. 教學上的價值

導入 AI Agent Workflow 的價值，不在於讓學生更快產出作業，而在於訓練可被檢查的工程流程。

學生可以學到：

- 如何把問題拆成 inspect、plan、patch、test、review 的步驟。
- 如何透過 Git diff review 觀察每一次修改。
- 如何用 test harness 驗證功能，而不是只相信輸出。
- 如何用 XML prompt 約束任務，避免 Agent 越界。
- 如何處理 context window overflow、memory 汙染與錯誤延續。
- 如何保留學習過程與決策紀錄，讓助教與教授可以評分。

## 7. 風險與控管方式

| 風險 | 控管方式 |
| --- | --- |
| 學生直接貼題目讓 AI 產出答案 | 要求提交 prompt、plan、diff、測試與反思紀錄。 |
| Agent 修改超出範圍 | 使用 XML 約束、AGENTS.md、限定檔案範圍，並執行 Git diff review。 |
| 產出看似正確但未驗證 | 要求 test harness 或可重現檢查步驟。 |
| 破壞既有架構或安全規則 | 事先定義 invariants，review 時逐項檢查。 |
| 忽略 edge cases | 評分要求列出並測試重要邊界情境。 |
| context window 或 memory 帶入錯誤資訊 | 要求學生說明使用的上下文、限制長期記憶依賴，必要時重建任務脈絡。 |
| Agent 直接影響主分支 | 不允許 Agent 自動 push main；使用 branch 與 pull request review。 |

## 8. 建議導入階段

| 階段 | 做法 | 目標 |
| --- | --- | --- |
| 第 1 階段：觀察 | 只允許 inspect-only 與 plan-only | 讓學生練習讀程式與拆任務。 |
| 第 2 階段：小範圍修改 | 允許限定檔案 patch | 建立 diff review 與 test harness 習慣。 |
| 第 3 階段：專題流程 | 加入 branch、PR、invariants、edge cases | 訓練可追溯的團隊協作。 |
| 第 4 階段：進階治理 | 討論 MCP、Skills、memory governance | 理解 Agent 系統設計與風險。 |

## 9. 評分方式建議

評分不宜只看最後產物，也應看學生是否掌握流程。

| 評分項目 | 建議觀察點 |
| --- | --- |
| 問題理解 | 是否能清楚定義需求、限制與成功條件。 |
| Prompt 與 XML 約束 | 是否能限制 Agent 任務範圍與輸出格式。 |
| Git diff review | 是否能解釋主要修改與拒絕不合理變更。 |
| Test harness | 是否有測試或可重現驗證步驟。 |
| Invariants 與 edge cases | 是否有列出並保護不可破壞規則與邊界情境。 |
| 反思與責任歸屬 | 是否能說明人的決策與 Agent 的協助範圍。 |

## 10. 給教授的結論

AI Agent Workflow 可以導入課堂，但前提是把它視為工程流程訓練，而不是作業代工工具。學生應被要求使用 Git、測試、review、prompt 約束與明確責任分工，證明自己能主導 Agent，而不是被 Agent 取代。

建議採取保守導入：先從 inspect-only 與 plan-only 開始，再逐步允許小範圍 patch。只要明確禁止 Agent 自動 push main，並要求 Git diff review、test harness、invariants 與 edge cases 紀錄，這套流程可以成為訓練學生工程判斷與工具治理能力的教材。
