## 1. 文件目的

本文件是給同學實際照著做的 AI Agent 使用指南。它不是工具廣告，也不是鼓勵把作業丟給 AI，而是教你如何把 Agent 放進可審查、可測試、可回退的工程 workflow。

你可以使用 Codex、Claude Code、VS Code Copilot Agent、Cline 或其他類似工具；本文件不強推單一工具。重點是流程：你要能控制 Agent 的任務邊界、檢查它改了什麼、測試結果是否正確，最後能自己解釋提交內容。

## 2. 使用 Agent 前的正確心態

AI Agent 不是替你完成作業的人。它比較像一個會讀檔、能提出建議、能產生草稿的協作者，但它可能誤解需求、漏掉 edge cases、改錯檔案、產生看似合理但未驗證的內容。

你仍然要負責：

| 你要負責的事 | 為什麼重要 |
| --- | --- |
| 理解需求 | 不懂題目就無法判斷 Agent 的答案是否正確。 |
| 定義架構 | Agent 可以給建議，但系統邊界與取捨要由人決定。 |
| 確認 invariants | 不能被破壞的規則要先講清楚。 |
| 檢查 edge cases | 很多錯誤出現在空值、邊界、權限、網路失敗等情境。 |
| 設計與執行測試 | 不能只相信 Agent 說「已完成」。 |
| 做 Git diff review | 你要看懂它實際改了什麼。 |
| 解釋結果 | 報告與口頭說明仍然是你的責任。 |

## 3. 不要直接叫 AI 幫我做完整專案

不要把任務寫成「幫我完成整個專案」。這會讓 Agent 一次跨過需求、架構、實作、測試與審查，最後你很難知道哪些內容可靠、哪些地方需要修正。

比較好的做法是把任務切小：

```text
先 inspect -> 再 plan -> 人類確認 -> 小範圍 patch -> 測試 -> review -> commit
```

如果你不知道怎麼開始，先要求 Agent 只讀取、不修改，並回報目前架構與下一步建議。

## 4. 標準 Agent Workflow：Inspect → Plan → Approve → Patch → Test → Review → Commit

```text
Inspect -> Plan -> Approve -> Patch -> Test -> Review -> Commit
   |         |         |         |       |        |          |
   |         |         |         |       |        |          +-- 由你提交紀錄
   |         |         |         |       |        +------------- 由你檢查 diff
   |         |         |         |       +---------------------- 執行測試或檢查
   |         |         |         +------------------------------ 只改批准範圍
   |         |         +---------------------------------------- 你確認計畫
   |         +-------------------------------------------------- Agent 提計畫
   +------------------------------------------------------------ Agent 先觀察
```

這個流程的目標是讓每一步都能被檢查。不要讓 Agent 跳過批准、測試或 review。

## 5. 每個階段同學應該做什麼

| 階段 | Agent 可以做 | 你應該做 |
| --- | --- | --- |
| Inspect | 讀檔、整理架構、列出問題 | 確認它讀到的內容是否正確。 |
| Plan | 提出修改步驟與風險 | 決定是否批准，必要時縮小範圍。 |
| Approve | 等待你的確認 | 明確說可以改哪些檔案、不能做什麼。 |
| Patch | 產生小範圍修改 | 不要離開；準備檢查 diff。 |
| Test | 執行或建議測試 | 確認測試有對準需求與 edge cases。 |
| Review | 摘要修改內容 | 自己閱讀 `git diff`，不要只看摘要。 |
| Commit | 可建議 commit message | 由你決定是否 commit，不要讓 Agent 自動 push main。 |

## 6. 如何寫一個好任務

好任務要具體、有限制、可驗證。

建議格式：

```text
目標：
請協助我處理 [具體問題]。

範圍：
只允許讀取/修改 [檔案或資料夾]。

限制：
不要刪除檔案。
不要執行 git push。
不要修改 main 分支。

流程：
請先 inspect 並提出 plan，等我批准後再修改。

驗證：
修改後請說明如何測試，並回報 diff summary。
```

## 7. 如何限制 Agent 修改範圍

限制範圍時要講清楚「可以」與「不可以」。

| 要限制的項目 | 寫法範例 |
| --- | --- |
| 可修改檔案 | 只允許修改 `src/login.ts` 與 `tests/login.test.ts`。 |
| 不可修改檔案 | 不要修改資料庫 schema、設定檔或 `README.md`。 |
| 不可執行動作 | 不要執行 `git push`，不要刪除檔案。 |
| 任務階段 | 先 inspect-only，不要修改。 |
| 測試要求 | 修改後請說明應執行哪些測試。 |

如果任務很大，先要求 Agent 拆成小任務，不要一次改完整專案。

## 8. 如何檢查 Agent 產出的程式或文件

不要只看 Agent 的回答。至少做三件事：

1. 看 `git diff`：確認它改了哪些行。
2. 跑測試或檢查：確認結果不是只在文字上看起來合理。
3. 自己解釋：確認你能說明為什麼這樣改。

檢查清單：

- 是否改到不該改的檔案？
- 是否破壞 invariants？
- 是否處理 edge cases？
- 是否留下未使用程式碼或假資料？
- 是否讓錯誤訊息消失，但沒有修到根因？
- 文件是否誇大或寫入未查證內容？

## 9. Git / GitHub 基礎概念

Git 是版本控制工具，用來記錄檔案變更、比較差異、回到舊版本。GitHub 是常見的遠端程式碼平台，用來同步 repository、開 pull request、做 review 與協作。

| 名詞 | 意思 |
| --- | --- |
| repository | 一個被 Git 管理的專案資料夾。 |
| commit | 一次被保存的變更紀錄。 |
| branch | 分支；用來隔離不同任務或實驗。 |
| remote | 遠端 repository，例如 GitHub 上的專案。 |
| pull | 從遠端抓取並合併更新。 |
| push | 把本地 commit 推到遠端。 |
| pull request | 請別人 review 你的 branch，再合併到主要分支。 |
| main / master | 常見的主要分支名稱；現在多數新專案使用 `main`。 |
| .gitignore | 告訴 Git 哪些檔案不要追蹤，例如暫存檔、密鑰、本地設定。 |

## 10. Git 最小日常流程

常用指令：

```bash
git status
git add
git commit
git diff
git log
git branch
git switch
git remote add origin
git push
git pull
```

基本意思：

| 指令 | 用途 |
| --- | --- |
| `git status` | 看目前哪些檔案被修改、是否在 branch 上。 |
| `git add <file>` | 把檔案加入下一次 commit。 |
| `git commit -m "message"` | 建立一筆變更紀錄。 |
| `git diff` | 看尚未 commit 的實際差異。 |
| `git log` | 看過去 commit 紀錄。 |
| `git branch` | 看目前有哪些 branch。 |
| `git switch <branch>` | 切換 branch。 |
| `git switch -c <branch>` | 建立並切換到新 branch。 |
| `git remote add origin <url>` | 設定 GitHub 遠端位置。 |
| `git push` | 推送到遠端。 |
| `git pull` | 從遠端拉回更新。 |

## 11. 使用 Agent 前為什麼要先 commit

使用 Agent 前先 commit，是為了留下安全點。Agent 可能改錯檔案、修改太多、或讓測試失敗；如果你事前有 commit，就能清楚比較「Agent 前」與「Agent 後」的差異。

同學第一次使用 Agent 前的安全流程：

```bash
git status
git add .
git commit -m "Save current work before agent changes"
git switch -c agent-task/example-task
```

如果 `git status` 顯示有你還不想提交的變更，先停下來整理，不要急著讓 Agent 修改。

## 12. Agent 修改後如何看 git diff

Agent 修改後，先不要 commit，也不要 push。先看：

```bash
git status
git diff
```

檢查重點：

- 改了哪些檔案？
- 是否改到任務以外的檔案？
- 是否刪掉重要內容？
- 是否有未解釋的大量重構？
- 測試或文件是否同步更新？

確認後再整理 commit：

```bash
git status
git diff
git add .
git commit -m "Apply agent-assisted changes"
```

## 13. 為什麼不要讓 Agent 直接 push main

`main` 是專案主要分支，通常代表穩定版本。讓 Agent 直接 push main 會跳過人類 review，也可能把錯誤、密鑰、破壞性修改或未測試內容推到團隊共用版本。

原則：

- Agent 可以協助修改，但不應自動 push main。
- 重要修改應先放在 branch。
- 由人看完 diff、跑完測試、確認後再 push。
- 團隊專題應用 pull request 做 review。

## 14. 如何用 branch 保護 main

建議每個 Agent 任務都開新 branch：

```bash
git switch -c agent-task/login-feature
```

命名方式可以清楚一點：

| 類型 | branch 範例 |
| --- | --- |
| 功能 | `agent-task/login-feature` |
| 修 bug | `agent-task/fix-login-error` |
| 文件 | `agent-task/update-docs` |
| 測試 | `agent-task/add-tests` |

完成後，把 branch push 到 GitHub，再開 pull request 給自己或隊友 review：

```bash
git push -u origin agent-task/login-feature
```

## 15. 如何把 Agent 的修改整理成可審查紀錄

一份可審查紀錄至少包含：

| 項目 | 內容 |
| --- | --- |
| 任務描述 | 你原本要解決什麼問題。 |
| Prompt | 你如何要求 Agent。 |
| Plan | Agent 提出的計畫，你批准了哪些部分。 |
| Diff summary | 主要改了哪些檔案與原因。 |
| 測試結果 | 執行了哪些測試或檢查，結果如何。 |
| 人類判斷 | 哪些地方是你確認、修改或拒絕的。 |

提交作業或專題報告時，不要只貼最終成果。要能說明你如何使用 Agent，以及你如何驗證它。

## 16. Context window 爆掉時怎麼辦

Context window 是 Agent 一次能處理的對話與檔案脈絡容量。對話太長、貼太多檔案、任務太大時，Agent 可能忘記前面內容、混淆需求或漏掉限制。

處理方式：

1. 停止繼續加新需求。
2. 請 Agent 先整理目前任務摘要。
3. 開新任務或新對話，只放必要背景。
4. 把任務切小，一次只處理一個檔案或一個功能。
5. 把穩定規則寫進 repo 文件，例如 `AGENTS.md` 或 `INVARIANTS.md`。
6. 重要結論重新檢查，不要相信舊對話記憶一定正確。

## 17. 本地模型與 memory 限制的基本概念

如果使用本地模型，例如透過本地推論工具接到 Agent，要理解基本限制。詳細安裝留給 `docs/04-建置指南.md`，安裝前請確認官方文件。

| 概念 | 意思 | 對同學的影響 |
| --- | --- | --- |
| RAM | 電腦系統記憶體 | 不足時模型可能跑不動或很慢。 |
| VRAM | 顯示卡記憶體 | 影響較大模型或較長 context 的執行能力。 |
| context window | 一次能放入的文字量 | 專案太大時要切任務，不要一次塞全部。 |
| 長期 memory | 工具可能保存的偏好或舊資訊 | 可能過期或錯誤，重要規則應寫在 repo 文件。 |

本地模型不等於一定安全或一定正確。你仍然要檢查 diff、測試結果與輸出內容。

## 18. 避免過度依賴 Agent 的方法

你可以讓 Agent 協助，但不要把判斷交出去。

建議做法：

- 先自己寫一版需求或解法，再請 Agent review。
- 要求 Agent 解釋取捨，而不是只給答案。
- 先讓 Agent inspect，不要一開始就 patch。
- 修改後自己看 diff，不只看 Agent 摘要。
- 每次提交前確認自己能口頭說明。
- 把不懂的地方列成問題，回頭查課本、文件或問助教。

## 19. 常見錯誤示範與改寫

| 錯誤 prompt | 問題 | 改寫 |
| --- | --- | --- |
| 幫我完成整個專案 | 範圍太大，無法審查，也容易跳過學習。 | 請先 inspect 專案，不要修改檔案，並回報目前架構與最小可行下一步。 |
| 幫我修 bug | 太模糊，容易亂改。 | 請根據以下錯誤訊息定位問題，只允許讀取相關檔案，先提出可能原因與驗證方式，不要先修改。 |
| 直接幫我 push 到 GitHub | 跳過 review，可能污染 main。 | 請只回報修改內容與建議 commit message，不要執行 git push。 |

更多可用改寫：

| 情境 | 建議 prompt |
| --- | --- |
| 不知道專案架構 | 請只 inspect 專案目錄與主要檔案，不要修改，整理架構、入口點與風險。 |
| 要加功能 | 請先提出 plan，列出會改哪些檔案、需要哪些測試，等我批准後再 patch。 |
| 要補測試 | 請根據目前功能列出正常案例、edge cases、錯誤案例，先不要修改。 |
| 要改文件 | 請只修改指定 Markdown 檔，使用繁體中文，不要加入未查證資訊。 |

## 20. 給同學的最小可行流程

每次要用 Agent 前，照這個最小流程做：

```bash
git status
git add .
git commit -m "Save current work before agent changes"
git switch -c agent-task/example-task
```

然後對 Agent 說：

```text
請先 inspect，不要修改檔案。
請回報目前狀況、可能風險、建議下一步。
等我批准後，才可以做小範圍修改。
```

Agent 修改後：

```bash
git status
git diff
git add .
git commit -m "Apply agent-assisted changes"
```

最後確認：

- 我看過 diff。
- 我知道改了哪些檔案。
- 我跑過測試或檢查。
- 我能解釋結果。
- 我沒有讓 Agent 直接 push main。
