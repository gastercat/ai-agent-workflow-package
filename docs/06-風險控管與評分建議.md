# 風險控管與評分建議

## 文件用途

提供 AI Agent Workflow 在課堂與專題中的風險控管框架、審查要求與評分 rubric。

## 目標讀者

- 教授
- 助教
- 同學

## 1. 文件目的

本文件提供教授、助教與專題組一套 AI Agent Workflow 的風險控管與評分建議。它不是工具介紹，也不是安裝指南，而是治理文件與評分參考。

導入 AI Agent 不代表放任學生用 AI 代工。合理的導入方式，是建立可審查、可驗證、可回退的工程流程，讓學生必須留下 prompt、plan、diff、測試結果、review 與反思。

核心主張：AI Agent 可以協助實作，但不能取代人類對架構、邊界、驗證、倫理與責任的判斷。課堂或專題評分也應從「最後程式碼」擴展到「問題定義、架構、約束、測試、審查、反思」。

## 2. 為什麼需要風險控管

Agent 能快速修改檔案，也可能快速放大錯誤。若沒有治理，常見問題包括：修改錯檔案、破壞既有功能、忽略邊界案例、捏造測試結果、洩漏 API key，或讓學生無法解釋自己的產出。

風險控管的目的不是阻止學生使用工具，而是要求使用工具時保留工程紀錄：

- 使用前知道目前 Git 狀態。
- 任務前定義 allowed_scope、forbidden_scope、invariants。
- 修改後做 `git diff` review。
- 用 test harness 驗證結果。
- 重要變更透過 branch 與 PR 審查。
- 報告中說明 Agent 做了什麼、人類檢查了什麼。

## 3. 風險總覽表

| 風險 | 風險說明 | 可能後果 | 控管方式 |
| --- | --- | --- | --- |
| Agent 修改錯檔案 | Prompt 範圍不清或 Agent 誤判相關檔案。 | 破壞無關功能、產生難以追蹤的變更。 | 使用 allowed_scope / forbidden_scope；修改後檢查 `git diff`。 |
| Agent 破壞既有功能 | 改動未考慮既有流程或 invariants。 | 舊功能失效、測試失敗、使用者資料錯誤。 | 任務前列 invariants；執行 regression test。 |
| Agent 幻覺 API 或不存在的指令 | Agent 產生看似合理但不存在的 API、套件或命令。 | 程式無法執行、安裝失敗、浪費除錯時間。 | 要求引用實際檔案與官方文件；測試指令必須實際執行或標明未執行。 |
| Agent 忽略 edge cases | 只處理 happy path。 | 空值、權限不足、網路失敗等情境出錯。 | 測試計畫必須包含 edge cases 與錯誤案例。 |
| Agent 自動 commit / push / deploy | Agent 直接影響版本庫或外部環境。 | main 被污染、錯誤部署、資料外洩。 | 禁止自動 push main、deploy；commit 與 push 由人類批准。 |
| context window 爆掉 | 專案太大或對話太長，Agent 漏掉前文。 | 忘記限制、重複錯誤、誤解需求。 | 切小任務；使用摘要與專案規則檔；重新 inspect。 |
| memory 記住錯誤資訊 | 長期記憶過期或記錯專案規則。 | 持續套用錯誤假設。 | 定期整理 `AGENTS.md`、`PROJECT_CONTEXT.md`、`INVARIANTS.md`。 |
| 本地模型能力不足 | 小模型或低資源環境無法理解大型 codebase。 | 產出品質不穩、推理慢、錯誤多。 | 先做小任務；不要把本地模型視為等同雲端 coding agent。 |
| MCP 權限過大 | Agent 可讀寫過多工具或資料源。 | 讀取私人資料、修改資料庫、誤觸外部服務。 | 最小權限；初學者少接 MCP server；優先 read-only。 |
| API key / `.env` 洩漏 | Agent 讀到或提交敏感檔案。 | 帳號外洩、費用損失、資料風險。 | `.gitignore` 排除；push 前檢查；不要貼敏感資訊給 Agent。 |
| 學生過度依賴 AI | 學生跳過需求理解與設計。 | 無法學會核心能力，評分失真。 | 要求提交 prompt、plan、diff、測試、反思。 |
| 學生無法解釋產出 | 學生只提交結果，無法說明過程。 | 無法確認學習成果與責任歸屬。 | 口頭或書面要求說明人類決策與 Agent 協助範圍。 |

## 4. Invariants：不可破壞的不變條件

Invariants 是系統在任何修改後都必須維持的規則。它們可能是資料格式、API contract、安全要求、分支規範或測試要求。

Agent 任務前要先寫 invariants，原因是 Agent 很容易為了完成局部任務而破壞全局規則。若沒有明確寫出不可破壞條件，review 時也難以判斷變更是否安全。

範例：

- API response schema 不可改變。
- 使用者資料格式不可改變。
- 測試不能被刪除。
- 登入失敗不可洩漏密碼。
- main branch 不可直接 push。
- 權限檢查不可移除。
- 錯誤處理不可只吞掉 exception。

建議把 invariants 放在：

- `INVARIANTS.md`
- `AGENTS.md`
- XML Prompt 的 `<invariants>`
- Pull request 檢查清單

## 5. Edge Cases：邊界條件與錯誤案例

Edge cases 是容易被忽略但實務上常出錯的情境。Agent 常能處理表面 happy path，但不一定會主動補齊邊界條件。

常見 edge cases：

| 類型 | 範例 |
| --- | --- |
| 空值 | 空字串、空陣列、沒有資料。 |
| 權限 | 未登入、權限不足、角色錯誤。 |
| 資料 | 重複資料、格式錯誤、極長輸入。 |
| 網路 | timeout、第三方 API 失敗、離線。 |
| 狀態 | 重複提交、並發請求、舊資料。 |
| 安全 | injection、資訊洩漏、錯誤訊息過度詳細。 |

評分時不應只看功能是否能在正常情境執行，也應看學生是否有列出並驗證重要 edge cases。

## 6. Permission Boundary：權限邊界

Permission boundary 是 Agent 能讀、能改、能執行的界線。邊界越清楚，越容易 review 與回退。

明確禁止：

- 不得自動 push main。
- 不得自動 deploy。
- 不得刪除資料。
- 不得修改 `.env`。
- 不得讀取無關私人檔案。
- 不得執行 `sudo` 或高風險命令，除非人工批准。
- 不得自行更改測試來掩蓋問題。
- 不得提交 API key、密碼、token 或私人資料。

建議每個任務 prompt 都包含：

- `allowed_scope`
- `forbidden_scope`
- `invariants`
- `verification`
- `stop_conditions`

## 7. Git / GitHub Review：branch、diff、PR、commit 紀錄

Git / GitHub 是 Agent Workflow 的審查基礎。

### 使用 Agent 前為什麼要先 commit

使用 Agent 前先 commit，可以建立可回退的安全點。若 Agent 修改失敗，可以清楚比較前後差異，或回到修改前狀態。

建議：

```bash
git status
git add .
git commit -m "Save current work before agent changes"
```

### 為什麼每個 Agent 任務要開 branch

每個 Agent 任務開 branch，可以隔離實驗與 main。

```bash
git switch -c agent-task/example-task
```

Branch 的好處：

- 失敗不會污染 main。
- 可以用 PR 做 review。
- 可以保留任務脈絡。
- 可以讓教授或助教追蹤修改過程。

### 為什麼要看 git diff

Agent 的摘要不等於真實修改。`git diff` 才能看到它實際改了哪些行。

Review 時檢查：

- 是否改到未批准檔案。
- 是否刪除測試或降低測試強度。
- 是否修改 `.env` 或敏感檔案。
- 是否破壞 invariants。
- 是否有不必要的大量重構。

### 為什麼 PR 比直接 push main 安全

Pull request 可以留下討論、review、測試結果與修改歷史。直接 push main 會跳過審查，也較難判斷學生是否理解變更。

### Commit message 如何描述 Agent-assisted changes

Commit message 應由學生撰寫，建議包含：

```text
Add login validation with agent assistance

- Used Agent to inspect login flow and propose patch
- Reviewed git diff manually
- Added tests for invalid password and empty input
- Verified no changes to API response schema
```

### 教授或助教如何追蹤理解過程

可以要求學生提交：

- 任務 prompt 或 XML Prompt。
- Agent plan。
- branch 名稱。
- 主要 commit。
- PR 或 diff 連結。
- 測試結果。
- 學生自己的 review notes。

## 8. Test Harness：測試作為 Agent 的現實約束

測試是 Agent 的現實約束。Agent 可以產生程式碼，但程式碼是否真的符合需求，需要透過測試與可重現檢查確認。

測試要求：

- 不只是測 happy path。
- 要包含 edge cases 與錯誤案例。
- 不允許為了通過測試而刪除測試。
- 不允許降低 assertion 強度。
- 測試結果不能捏造。
- 如果無法執行測試，必須說明原因與替代驗證方式。

Test harness 可以包含：

- 單元測試。
- 整合測試。
- lint / type check。
- 手動重現步驟。
- API request / response 範例。
- 截圖或操作紀錄，若課程允許。

## 9. Context Window Overflow：上下文爆掉的風險

Context window overflow 指 Agent 可處理的上下文容量不足。常見原因是對話太長、一次貼太多檔案、專案太大、或任務沒有切分。

風險：

- Agent 忘記先前限制。
- Agent 漏讀重要檔案。
- Agent 混淆舊需求與新需求。
- Agent 把推測當成事實。

控管方式：

- 大任務切成 inspect、plan、patch、test、review。
- 只提供本次任務需要的檔案。
- 要求 Agent 說明它實際看過哪些檔案。
- 把長期規則寫入 `AGENTS.md`。
- 把專案背景寫入 `PROJECT_CONTEXT.md`。
- 把不可破壞規則寫入 `INVARIANTS.md`。

## 10. Memory Governance：長期記憶治理

Memory 在 Agent Workflow 中至少分成三類：

| 類型 | 說明 | 風險 |
| --- | --- | --- |
| 硬體記憶體 | RAM / VRAM，影響本地模型能否執行。 | 資源不足導致慢、失敗或品質不穩。 |
| context window | 單次對話可放入的內容量。 | 太長會漏脈絡，太短會缺資訊。 |
| 長期專案記憶 | Agent 或工具保存的偏好、規則、過往決策。 | 可能過期、錯誤或與新需求衝突。 |

治理原則：

- 長期記憶不能當成事實來源。
- 重要規則要放在 repo 檔案中。
- `AGENTS.md`、`PROJECT_CONTEXT.md`、`INVARIANTS.md` 可以作為外部記憶。
- 這些文件需要定期整理與修正。
- 任務開始前仍應 inspect 現況。

## 11. Local Model Memory Limits：本地模型的 RAM / VRAM / context 限制

本地模型可用於課堂或專題實驗，但不應被誇大。它可能比較便宜、較容易自管資料，但不一定比較強，也不一定適合大型 codebase。

注意事項：

| 項目 | 說明 |
| --- | --- |
| RAM | 系統記憶體不足會讓模型載入失敗或非常慢。 |
| VRAM | GPU 記憶體影響可載入模型大小與推理速度。 |
| 模型大小 | 大模型通常需要更多資源，但不保證更適合任務。 |
| 量化 | 可降低記憶體需求，但可能影響品質。 |
| context window | 長上下文會消耗更多資源，也可能降低穩定性。 |
| 多 Agent 同時運作 | 會增加 CPU/GPU、RAM/VRAM 與 I/O 負載。 |

大型 codebase 應使用外部記憶檔案，例如：

- `AGENTS.md`
- `PROJECT_CONTEXT.md`
- `ARCHITECTURE.md`
- `INVARIANTS.md`

## 12. 敏感資料、API key 與 .env

敏感資料治理是最低限度要求。

禁止：

- 把 API key 貼給 Agent。
- 把 `.env` 提交到 Git。
- 把密碼、token、私人資料放進 prompt。
- 讓 Agent 讀取無關私人目錄。
- 讓 Agent 將敏感資訊寫入文件或 commit。

建議：

- 使用 `.gitignore` 排除 `.env`。
- push 前執行 `git status` 與 `git diff`。
- 若疑似 key 外洩，立即撤銷並重建。
- 課堂示範使用 dummy key 或範例設定。

## 13. MCP 與工具權限風險

MCP 可以讓 Agent 連接 filesystem、Git、GitHub、database、browser、docs 等外部工具。它讓 Agent 更有能力，也讓權限風險變大。

控管原則：

- MCP 不是越多越好。
- 初學者不要一次接太多 MCP server。
- 權限要最小化。
- 可 read-only 就不要給 write。
- 不要把 production database 接給課堂練習 Agent。
- browser、GitHub、database 類工具要特別謹慎。
- MCP 設定變更也應納入 review。

## 14. XML Prompt 約束的作用與限制

XML Prompt 可用來明確分區：

- `allowed_scope`
- `forbidden_scope`
- `invariants`
- `edge_cases`
- `verification`
- `output_format`
- `stop_conditions`

作用：

- 降低任務邊界模糊。
- 讓禁止事項更清楚。
- 方便團隊共用。
- 方便與 `AGENTS.md`、任務卡、PR template 搭配。

限制：

- XML 不能保證 Agent 一定遵守。
- XML 不能取代測試。
- XML 不能取代 `git diff` review。
- XML 不能取代人類判斷。
- Prompt 太長仍然可能受 context window 限制。

## 15. Agent 使用紀錄應該保留什麼

建議每個 Agent 任務保留：

| 紀錄 | 用途 |
| --- | --- |
| 任務描述 | 說明要解決什麼問題。 |
| Prompt / XML Prompt | 證明任務範圍與限制。 |
| Agent plan | 顯示修改前的設計與風險。 |
| Branch 名稱 | 追蹤任務隔離方式。 |
| Diff summary | 說明實際變更。 |
| 測試結果 | 證明驗證過程。 |
| Review notes | 說明人類如何檢查與修正。 |
| 反思 | 說明限制、學到什麼、哪些地方仍需確認。 |

## 16. 教授與助教可以如何審查

教授與助教不需要重跑每一步，但可以抽查流程是否可信。

審查方式：

- 看學生是否提交 prompt / XML Prompt。
- 看 branch 與 commit 是否對應任務。
- 看 `git diff` 是否合理。
- 看 PR 討論是否包含 review。
- 看測試結果是否可重現。
- 口頭詢問學生解釋關鍵修改。
- 抽查 invariants 是否被破壞。
- 抽查 edge cases 是否被測試。
- 檢查是否有 `.env`、API key 或私人資料。

## 17. 評分 Rubric

| 評分項目 | 評分重點 | 優秀表現 | 需改進表現 |
| --- | --- | --- | --- |
| 問題定義 | 是否理解需求、限制與成功條件。 | 能清楚拆解問題，說明成功條件與限制。 | 只貼題目給 Agent，無法說明需求。 |
| 架構設計 | 是否有人類主導架構與模組邊界。 | 能說明架構取捨、資料流與責任分工。 | 完全依賴 Agent 決定架構。 |
| Prompt / XML 約束品質 | 是否定義 allowed_scope、forbidden_scope、verification。 | Prompt 清楚、可審查，包含停止條件。 | Prompt 模糊，例如「幫我完成」。 |
| Invariants 設計 | 是否列出不可破壞條件。 | 有明確 API、資料、安全、分支規則。 | 沒有 invariants，或只寫空泛原則。 |
| Edge cases 設計 | 是否考慮邊界與錯誤案例。 | 覆蓋空值、權限、錯誤輸入、外部失敗。 | 只測 happy path。 |
| 測試設計 | 是否有 test harness 或可重現驗證。 | 測試可執行，結果清楚，未捏造。 | 沒有測試，或刪測試讓程式通過。 |
| Agent 使用流程 | 是否遵守 inspect、plan、approve、patch、test、review。 | 每階段有紀錄，修改前有批准。 | Agent 直接大範圍修改。 |
| Git / GitHub 紀錄 | 是否使用 branch、commit、diff、PR。 | 任務 branch 清楚，commit message 說明 Agent-assisted changes。 | 直接 push main，或沒有可追溯紀錄。 |
| Review 與修正能力 | 是否能看懂並修正 Agent 產出。 | 能拒絕不合理修改並說明原因。 | 只接受 Agent 輸出，無人工判斷。 |
| 最終成果 | 功能、文件或測試是否達成目標。 | 成果可執行、可讀、符合需求。 | 表面完成但無法驗證或破壞舊功能。 |
| 反思與限制說明 | 是否能說明 AI 協助範圍與限制。 | 清楚區分人類決策、Agent 協助與未確認風險。 | 無法說明哪些內容由誰決定。 |

## 18. 最小導入規範

課堂或專題若要允許 Agent，建議至少要求以下 checklist：

- [ ] 使用 Agent 前先 commit。
- [ ] 每個任務開 branch。
- [ ] 任務 prompt 必須包含 allowed_scope / forbidden_scope / invariants。
- [ ] Agent 修改後必須看 `git diff`。
- [ ] 必須保留測試結果。
- [ ] 禁止自動 push main。
- [ ] 禁止自動 deploy。
- [ ] 禁止提交 API key / `.env`。
- [ ] 報告中要說明 Agent 做了什麼、人類 review 了什麼。
- [ ] 如果測試無法執行，必須說明原因與替代驗證方式。

## 19. 給教授的建議結論

AI Agent 可以被納入課堂與專題，但應作為工程流程的一部分，而不是作業代工工具。導入時不宜只評最後程式碼，應評估學生如何定義問題、限制 Agent、設計測試、檢查 diff、保護 branch、處理 edge cases，並反思工具限制。

建議採取保守導入：先要求 inspect-only 與 plan-only，再允許小範圍 patch。只要明確要求 Git / GitHub review、test harness、invariants、permission boundary、memory governance 與使用紀錄，Agent Workflow 可以成為訓練學生工程判斷與責任意識的教材。
