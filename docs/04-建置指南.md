## 1. 文件目的

本文件提供同學與專題小組一份可實際照做的 AI Agent 工具建置指南。重點是把工具放進安全、可審查、可測試的 workflow，而不是比較誰是唯一最佳工具。

本文會介紹多條工具路線：

- VS Code Copilot Agent
- Codex CLI
- Claude Code
- Cline
- Continue
- Ollama + 本地開源模型
- Open WebUI 作為實驗室共用入口

涉及安裝方式、帳號、授權、模型名稱、指令或平台支援時，安裝前請確認官方文件。不要依賴過期教學文或未確認的下載連結。

## 2. 選擇工具前先判斷自己的需求

先不要急著安裝所有工具。請先判斷你要解決什麼問題。

| 需求 | 較適合的方向 |
| --- | --- |
| 你主要使用 VS Code | VS Code Copilot Agent、Cline、Continue |
| 你熟悉 terminal | Codex CLI、Claude Code |
| 你想用本地模型 | Cline、Continue、Ollama |
| 你要課堂 demo | VS Code Copilot Agent、Open WebUI、Cline |
| 你要做專題開發 | Codex CLI、Claude Code、Cline、VS Code Copilot Agent |
| 你想研究 MCP | Cline、Claude Code、支援 MCP 的 Agent 工具 |
| 你要多人共用聊天入口 | Open WebUI |

基本原則：

- 新手先選一條路線，不要一次裝太多。
- 先確認 Git 工作流，再讓 Agent 修改檔案。
- 先做 inspect-only 測試，再允許 patch。
- 不要把本地模型當成雲端 coding agent 的等價替代品。

## 3. 工具路線總覽

| 路線 | 工具 | 主要介面 | 適合對象 |
| --- | --- | --- | --- |
| A | VS Code Copilot Agent | VS Code | 初學者、VS Code 使用者、課堂展示 |
| B | Codex CLI | Terminal | CLI 使用者、需要 repo 內工作流的人 |

## 4. 路線 A：VS Code Copilot Agent

### 適合對象

- 已經使用 VS Code 的同學。
- 第一次接觸 Agent，需要視覺化介面的人。
- 課堂 demo，需要清楚展示檔案修改與 review 的情境。

### 優點

- 直接在 editor 內使用，容易看到檔案與變更。
- 適合搭配 VS Code 的 Git UI、terminal、測試工具。
- 對初學者相對容易上手。

### 限制

- 需要 GitHub 帳號與對應服務權限。
- 組織或學校帳號可能限制 Copilot 或 agent 功能。
- 可能一次修改多個檔案，仍然必須做 Git diff review。

### 基本建置概念

1. 安裝或更新 VS Code。
2. 登入 GitHub 帳號。
3. 確認 Copilot / Agent 功能是否可用。
4. 開啟專案資料夾。
5. 先確認 `git status`，再執行 inspect-only 任務。

安裝前請確認 VS Code 與 GitHub Copilot 官方文件。

### 第一次測試任務

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- 不要一開始就要求「完成整個專案」。
- 不要讓 Agent 自動接受所有修改。
- 修改後要看 VS Code diff 或 `git diff`。

## 5. 路線 B：Codex CLI

### 適合對象

- 熟悉 terminal 的同學。
- 想在 repo 中進行 inspect、patch、test 的使用者。
- 想練習 CLI 型 Agent workflow 的專題小組。

### 優點

- 適合與 Git、測試指令、命令列工具結合。
- 容易把工作流拆成 inspect、plan、patch、test。
- 適合訓練「修改前先看狀態、修改後看 diff」的習慣。

### 限制

- 需要理解 terminal、檔案路徑、權限與命令輸出。
- 帳號、模型、授權與安裝方式可能變動。
- 執行命令前要確認是否會修改檔案或外部服務。

### 基本建置概念

1. 確認電腦已安裝 Git 與基本 command line 工具。
2. 依官方文件安裝 Codex CLI。
3. 完成帳號或 API 驗證設定。
4. 在專案根目錄啟動工具。
5. 先執行 inspect-only 任務，不要直接 patch。

安裝前請確認 OpenAI Codex / Codex CLI 官方文件。

### 第一次測試任務

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- 不要讓 CLI Agent 自動執行你看不懂的命令。
- 要求它先提出 plan，再由你批准。
- 不要執行 `git push`，除非你已經手動 review 並確認。

## 6. 路線 C：Claude Code

### 適合對象

- 熟悉 terminal 的同學。
- 想使用命令列 agentic coding tool 的使用者。
- 需要把讀檔、修改、測試、說明整合在一個互動流程中的專題小組。

### 優點

- 適合在 terminal 中處理 codebase。
- 適合做 bug 定位、重構建議、測試補強與文件整理。
- 可用於示範「先 plan，再改」的工作方式。

### 限制

- 需要帳號、授權與環境設定。
- 可能會建議執行 shell command，必須由人審查。
- 不是所有任務都適合自動修改。

### 基本建置概念

1. 確認官方文件列出的必要環境。
2. 依官方文件安裝 Claude Code。
3. 完成登入或 API 設定。
4. 在專案根目錄開啟工具。
5. 先要求 inspect-only，再進入 plan。

安裝前請確認 Anthropic Claude Code 官方文件。

### 第一次測試任務

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- 不要把 terminal 權限全部交給 Agent。
- 對檔案修改與命令執行維持人工批准。
- 修改後一定要檢查 `git diff` 與測試結果。

## 7. 路線 D：Cline

### 適合對象

- 使用 VS Code 或相容 editor 的同學。
- 想比較不同模型 provider 的使用者。
- 想嘗試本地模型與 MCP 的同學。

### 優點

- 可在 editor 中查看 Agent 行動。
- 可搭配不同模型或 provider。
- 適合示範每一步都需要批准的 Agent workflow。

### 限制

- 模型費用、API key、權限與工具調用要自己管理。
- 本地模型品質與速度不一定足夠處理大型專案。
- MCP 或工具權限設定不當會增加風險。

### 基本建置概念

1. 安裝支援的 editor。
2. 依官方文件安裝 Cline extension 或 CLI。
3. 選擇模型來源，例如雲端 provider 或本地模型。
4. 設定 API key 或本地模型 endpoint。
5. 先用 inspect-only 任務確認能讀到專案。

安裝前請確認 Cline 官方文件。

### 第一次測試任務

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- API key 不要提交到 Git。
- 不要一次開啟過多工具權限。
- 如果使用本地模型，先用小專案測試速度與品質。

## 8. 路線 E：Continue

### 適合對象

- 想在 IDE 中使用 AI assistant 的同學。
- 想配置不同模型來源的人。
- 想把 code completion、chat、refactor 融入 editor 的使用者。

### 優點

- 適合 editor 內輔助開發。
- 可配置不同模型與使用方式。
- 適合練習小範圍修改與文件整理。

### 限制

- 設定彈性高，也代表需要理解模型與 provider。
- 不同模型的 coding 能力差異明顯。
- 仍需 Git diff review，不可只相信建議。

### 基本建置概念

1. 安裝支援的 IDE 或 editor。
2. 依官方文件安裝 Continue。
3. 選擇模型 provider 或本地模型。
4. 設定必要的 key、endpoint 或模型。
5. 先做小範圍任務，確認輸出品質。

安裝前請確認 Continue 官方文件。

### 第一次測試任務

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- 不要同時接太多模型與工具，先讓設定簡單。
- 如果接本地模型，要注意 context window 與速度。
- 修改後仍要看 `git diff`。

## 9. 路線 F：Ollama + 本地開源模型

### 適合對象

- 想實驗 local model 的同學。
- 想降低部分資料外傳疑慮的專題小組。
- 想理解 RAM、VRAM、量化模型與 context window 限制的人。

### 優點

- 可在本機或自管伺服器執行模型。
- 適合課堂實驗本地模型行為。
- 可搭配 Cline、Continue 或其他支援 local endpoint 的工具。

### 限制

- 本地模型可能比較便宜，但不一定比較強。
- coding 能力、長上下文能力與工具使用能力依模型差異很大。
- 模型越大，通常需要更多 RAM / VRAM，也可能更慢。
- 不應把本地模型視為等同 Codex CLI 或 Claude Code。

### 基本建置概念

1. 依官方文件安裝 Ollama 或類似 local inference 工具。
2. 選擇適合硬體的模型大小。
3. 先用短 prompt 測試模型能否正常回應。
4. 再接到 Cline、Continue 或其他支援本地模型的 Agent 工具。
5. 先做 inspect-only，不要讓本地模型直接大範圍修改。

安裝前請確認 Ollama 官方文件與模型來源說明。

### 第一次測試任務

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- 小模型不一定能理解大型 codebase。
- 長 context 會增加記憶體壓力。
- 同時開多個 Agent 或多個模型會提高 RAM、VRAM 與 CPU/GPU 負載。
- 大型 codebase 需要外部記憶檔案，例如 `AGENTS.md`、`PROJECT_CONTEXT.md`、`ARCHITECTURE.md`、`INVARIANTS.md`。

## 10. 路線 G：Open WebUI 作為實驗室共用入口

### 適合對象

- 實驗室或課堂想提供共用 web 入口。
- 想把多個模型或本地服務集中管理的人。
- 不想讓每位同學都直接操作底層模型服務的情境。

### 優點

- 使用 web 介面，對初學者較容易。
- 可作為本地模型或伺服器模型的共用入口。
- 適合課堂展示不同模型輸出差異。

### 限制

- 需要管理伺服器、帳號、權限與資料保存。
- 不等同完整 coding agent；是否能改檔案取決於整合方式。
- 若開放多人使用，要注意隱私、紀錄與資源競爭。

### 基本建置概念

1. 先確認要連接的模型服務，例如 Ollama 或其他 backend。
2. 依官方文件部署 Open WebUI。
3. 設定使用者、模型連線與存取權限。
4. 先用一般問答測試，再測試專案摘要任務。
5. 不要直接提供過大的專案或敏感資料。

安裝前請確認 Open WebUI 官方文件。

### 第一次測試任務

```text
請先 inspect 以下我提供的專案摘要，不要假設你能讀取本機檔案。
請回報：
1. 你理解的專案結構
2. 可能缺少的資訊
3. 可能的下一步
4. 你需要我確認的地方
```

### 注意事項

- Web UI 不一定能直接讀你的 repo。
- 不要貼 API key、`.env`、個資或未授權資料。
- 若作為共用入口，需有帳號與資料保存規則。

## 11. Git / GitHub remote 設定

Git / GitHub 是 Agent Workflow 的安全底線。讓 Agent 修改前，請先確認專案是否已經被 Git 管理。

如果是新專案，可以使用以下概念流程：

```bash
git init
git status
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin <你的 GitHub repo URL>
git push -u origin main
```

如果已經有 repo，不要重複 `git init`。先執行：

```bash
git status
git remote -v
git branch
```

推送前請確認：

- 沒有 API key。
- 沒有 `.env`。
- 沒有私人資料、作業答案外洩資料或未授權資料。
- `.gitignore` 已排除不該提交的檔案。

`.gitignore` 的作用是告訴 Git 不要追蹤某些檔案，例如：

```text
.env
node_modules/
dist/
.DS_Store
*.log
```

為什麼不要讓 Agent 自動 push：

- Agent 可能把錯誤或敏感資料推到遠端。
- Agent 可能跳過 review。
- Agent 可能直接污染 main。

為什麼建議用 branch 做 Agent 任務：

- 可以隔離實驗。
- 可以用 pull request review。
- 如果修改失敗，main 不會被破壞。

範例：

```bash
git switch -c agent-task/setup-agent-guide
# 讓 Agent 在 branch 上工作
git status
git diff
git add .
git commit -m "Document agent setup workflow"
git push -u origin agent-task/setup-agent-guide
```

## 12. 專案內 AGENTS.md 的建置方式

`AGENTS.md` 是放在 repo 裡給 Agent 讀的專案規則。它應該寫清楚 Agent 可以做什麼、不能做什麼、修改後要如何回報。

建議內容：

```markdown
# AGENTS.md

## Project Goal
說明專案目標。

## Language
使用繁體中文。

## Working Rules
- 先 inspect，再 plan。
- 未經批准不要修改檔案。
- 不要刪除檔案。
- 不要執行 git push。
- 修改後回報 diff summary。

## Verification
- 說明應執行哪些測試或檢查。
```

大型專案可再加入：

- `PROJECT_CONTEXT.md`
- `ARCHITECTURE.md`
- `INVARIANTS.md`
- `TASK_TEMPLATE.xml`

這些檔案可以降低 Agent 遺漏背景、破壞 invariants 或被舊 memory 誤導的機率。

## 13. MCP 基礎建置概念

MCP 是讓 Agent 連接外部工具、資料或服務的協定。它可以讓 Agent 接觸 filesystem、Git、GitHub、database、browser、docs 等資源。

MCP 不是越多越好。每多接一個 MCP server，就多一個權限與資料風險。

| MCP 類型 | 可能用途 | 風險 |
| --- | --- | --- |
| filesystem | 讀寫本機檔案 | 可能讀到或改到不該碰的檔案。 |
| Git / GitHub | 查 commit、issue、PR | 可能影響 repo 或暴露私有資訊。 |
| database | 查詢資料庫 | 可能讀到個資或修改資料。 |
| browser | 操作網頁 | 可能登入錯誤帳號或觸發外部動作。 |
| docs | 查官方文件 | 來源與版本仍要確認。 |

建置原則：

- 初學者先不要一次接太多 MCP server。
- 權限要最小化，只開任務需要的 server。
- 能 read-only 就不要開 write。
- 不要把 production database 接給初學者 Agent。
- 設定後先做 inspect-only 測試。

涉及 MCP server 安裝、設定檔格式與支援工具時，請確認 MCP 與各工具官方文件。

## 14. 本地模型硬體需求與 memory 注意事項

本地模型部署前，先理解硬體與 memory 限制。

| 概念 | 說明 |
| --- | --- |
| RAM | 系統記憶體；模型、工具、IDE、瀏覽器都會占用。 |
| VRAM | GPU 記憶體；影響模型是否能放進顯示卡與推理速度。 |
| 量化模型 | 用較低精度降低模型大小與記憶體需求，但可能影響品質。 |
| 模型大小 | 通常越大越吃資源，也不保證越適合你的任務。 |
| context window | 一次能放入的文字量；越長通常越耗資源。 |
| 推理速度 | 受 CPU、GPU、RAM、VRAM、模型大小與量化方式影響。 |
| 多 Agent 同時運作 | 會增加 CPU/GPU、RAM/VRAM、API 或本地服務負載。 |

務實判斷：

- 本地模型可能比較便宜，但不一定比較強。
- 小模型適合摘要、簡單問答或小檔案任務；大型 codebase 仍可能吃力。
- 不要把整個 repo 塞進 context window。
- 大型 codebase 需要外部記憶檔案：`AGENTS.md`、`PROJECT_CONTEXT.md`、`ARCHITECTURE.md`、`INVARIANTS.md`。
- 本地模型輸出仍要經過 Git diff review 與 test harness。

## 15. 第一次測試任務

不管你選哪一條工具路線，第一次都建議用 inspect-only。

```text
請先 inspect 這個專案，不要修改任何檔案。
請回報：
1. 專案目錄結構
2. 主要檔案用途
3. 可能的下一步
4. 你需要我確認的地方
```

如果 Agent 回答合理，再進入 plan-only：

```text
請根據剛才的 inspect 結果提出修改計畫。
不要修改檔案。
請列出：
1. 建議修改的檔案
2. 每個檔案的修改目的
3. 風險
4. 測試方式
```

## 16. 常見問題與排錯

| 問題 | 可能原因 | 處理方式 |
| --- | --- | --- |
| Agent 找不到專案檔案 | 沒有在專案根目錄啟動、workspace 開錯 | 回到專案根目錄，確認 `pwd`、`ls`、VS Code workspace。 |
| API key 設定錯誤 | key 沒設、設錯位置、權限不足 | 重新依官方文件設定，確認不要提交到 Git。 |
| 本地模型跑很慢 | 模型太大、RAM/VRAM 不足、CPU 推理 | 換小模型、降低 context、關閉其他模型或工具。 |
| context window 不夠 | 貼太多檔案、任務太大 | 切小任務，使用摘要與專案規則檔。 |
| Agent 修改太多檔案 | prompt 範圍太大 | 停止修改，檢查 `git diff`，下一次限制檔案範圍。 |
| `git push` 被拒絕 | 遠端有新 commit、權限不足、branch 不對 | 先看錯誤訊息，必要時 `git pull`，不要盲目 force push。 |
| 中文檔名在 terminal 顯示 escape | Git 或 terminal 顯示設定問題 | 先確認實際檔案未損壞；可查 Git quote path 顯示設定。 |
| MCP server 權限太大 | 接了過多工具或給了 write 權限 | 關閉不必要 server，改 read-only 或縮小資料範圍。 |
| VS Code extension 權限不清楚 | extension 可讀寫 workspace 或執行命令 | 查看 extension 文件與設定，先在測試 repo 試用。 |

## 17. 安全注意事項

- 不要把 API key、`.env`、密碼、token、私人資料貼給 Agent。
- 不要讓 Agent 自動 push main。
- 不要讓 Agent 操作 production database。
- 不要安裝來源不明的 extension、MCP server 或模型。
- 不要把本地模型輸出視為一定安全或正確。
- 每次修改後都要看 `git diff`。
- 重要任務要能回退：先 commit，再讓 Agent 修改。

## 18. 建議工具選擇表

| 你目前的情境 | 建議先試 |
| --- | --- |
| 完全新手、使用 VS Code | VS Code Copilot Agent 或 Cline |
| 熟悉 CLI、想練 Agent workflow | Codex CLI 或 Claude Code |
| 想在 IDE 中接不同模型 | Continue 或 Cline |
| 想研究本地模型 | Ollama + Cline / Continue |
| 想做實驗室共用入口 | Open WebUI |
| 想研究工具連接與資料源 | MCP + 支援 MCP 的 Agent 工具 |

選擇工具時請記住：工具只是入口，真正要建立的是 Human-led Architecture、Git diff review、test harness、permission boundary 與可追溯紀錄。
